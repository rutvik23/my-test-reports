Test Plan: PR #8787 — Make deleteCLISecrets non-fatal during cluster destroy PASS

Repo: openshift/hypershift
PR: https://github.com/openshift/hypershift/pull/8787
JIRA: OCPBUGS-86669

PR Summary

hcp destroy cluster (the HyperShift destroy CLI flow) sets the openshift.io/destroy-cluster finalizer on the HostedCluster, then runs, in order: 

setFinalizer() → destroyPlatformSpecifics() → deleteCLISecrets() → removeFinalizer()

deleteCLISecrets calls client.DeleteAllOf on Secrets, which requires the deletecollection RBAC verb — a distinct verb from list/get/delete. If the credentials running the CLI lack that verb, the call errors, DestroyCluster returns early, and removeFinalizer never runs — the HostedCluster is stuck forever with a deletionTimestamp and the destroy finalizer still attached.

Changes:

  • cmd/cluster/core/destroy.go: deleteCLISecrets errors are now logged and swallowed instead of aborting the destroy flow (secrets are owner-ref'd to the HostedCluster and get garbage collected, or removed when the operator deletes the control-plane namespace). Also introduces a package-level newClient var (defaults to util.GetClientWithKubeconfig) so tests can inject a client with interceptor-based error injection.
  • cmd/util/client.go: the fake client used when FAKE_CLIENT=true now registers hyperapi.Scheme (superset of the default k8s scheme) instead of the bare default scheme, so tests can use HyperShift types like HostedCluster.
  • cmd/cluster/core/destroy_test.go: adds two new unit tests — one exercising the full destroy flow with a real (non-nil) HostedCluster and asserting the destroy finalizer lifecycle, and one that forces deleteCLISecrets to fail (via an interceptor returning 403 Forbidden on DeleteAllOf for Secrets) and asserts DestroyCluster still returns no error.

Key fact established during investigation

Built-in admin/edit ClusterRoles and cluster-admin already include deletecollection on secrets, so a normal admin kubeconfig against a freshly-installed HostedCluster will never hit this bug. Reproducing it requires deliberately creating a Role that grants secret delete but omits deletecollection. There is no shortcut around this setup — the walkthrough below builds it explicitly.

Repo state (verified 2026-07-21, local clone of openshift/hypershift):

  • main (HEAD 1aca754107) still has the unfixed code — cmd/cluster/core/destroy.go:167 returns fatally on any deleteCLISecrets error. This is your pre-fix binary; no need to check out an older commit.
  • fix/OCPBUGS-86669-destroy-finalizer (commit e9883b43f2, built directly on top of main at f69e73409f) has the fix. This is your post-fix binary.

Prerequisites

  • A HyperShift management cluster you're logged into as cluster-admin via oc, with oc and go available locally, and this repo cloned.
  • Ability to create/delete a HostedCluster (any supported platform — AWS is easiest to reproduce the RBAC scenario against, but the core logic is platform-agnostic). Use whichever cluster-creation flow you normally use in this environment (e.g. the hcp:hcp-create-aws skill) for step 2 below.

Step-by-step reproduction walkthrough

Run these in order. Steps 1–11 reproduce the bug on the unfixed code; steps 12–16 verify the fix resolves it, reusing the same stuck cluster and restricted kubeconfig.

0. Set variables

export CLUSTER_NAME=destroy-repro-1
export CLUSTER_NS=clusters                # namespace holding the HostedCluster
export PLATFORM=aws                       # or your platform of choice
export SA_NAME=hcp-destroy-no-deletecollection

1. Confirm you're on the pre-fix code and build the CLI

cd hypershift
git status                                # should be clean, on main
git log --oneline -1                      # sanity-check HEAD
grep -n -A3 "if err = deleteCLISecrets" cmd/cluster/core/destroy.go
# expect to see: return err   (no logging/swallow — this confirms pre-fix)

make product-cli                          # builds ./bin/hcp
./bin/hcp version

2. Create a fresh HostedCluster and wait for it to become Available

# Use your normal cluster-creation flow for $PLATFORM, naming the cluster $CLUSTER_NAME
# in namespace $CLUSTER_NS. Then wait for Available:
oc get hostedcluster "${CLUSTER_NAME}" -n "${CLUSTER_NS}" \
  -o jsonpath='{.status.conditions[?(@.type=="Available")].status}{"\n"}'

export INFRA_ID=$(oc get hostedcluster "${CLUSTER_NAME}" -n "${CLUSTER_NS}" \
  -o jsonpath='{.spec.infraID}')
echo "INFRA_ID=${INFRA_ID}"

3. Note the CLI-generated secrets before destroy (for later GC verification)

oc get secrets -n "${CLUSTER_NS}" -l hypershift.openshift.io/auto-created-for-infra="${INFRA_ID}" \
  -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.metadata.ownerReferences[*].kind}/{.metadata.ownerReferences[*].name}{"\n"}{end}'

4. Create the restrictive Role (secrets delete allowed, deletecollection withheld)

cat <<EOF | oc apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: ${SA_NAME}
  namespace: ${CLUSTER_NS}
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"] # no "deletecollection"
- apiGroups: ["hypershift.openshift.io"]
  resources: ["hostedclusters"]
  verbs: ["get", "list", "watch", "update", "patch", "delete"]
EOF

5. Create the ServiceAccount and bind it to the restrictive Role

oc create serviceaccount "${SA_NAME}" -n "${CLUSTER_NS}"

oc create rolebinding "${SA_NAME}" \
  --role="${SA_NAME}" \
  --serviceaccount="${CLUSTER_NS}:${SA_NAME}" \
  -n "${CLUSTER_NS}"

6. Generate a kubeconfig for the restricted ServiceAccount

TOKEN=$(oc create token "${SA_NAME}" -n "${CLUSTER_NS}" --duration=1h)
SERVER=$(oc whoami --show-server)

oc config set-cluster restricted-cluster --server="${SERVER}" --insecure-skip-tls-verify=true \
  --kubeconfig=/tmp/restricted-kubeconfig
oc config set-credentials "${SA_NAME}" --token="${TOKEN}" \
  --kubeconfig=/tmp/restricted-kubeconfig
oc config set-context restricted --cluster=restricted-cluster --user="${SA_NAME}" \
  --namespace="${CLUSTER_NS}" --kubeconfig=/tmp/restricted-kubeconfig
oc config use-context restricted --kubeconfig=/tmp/restricted-kubeconfig

7. Sanity-check the restriction is actually in place

# should succeed (delete is allowed)
oc auth can-i delete secrets -n "${CLUSTER_NS}" --kubeconfig=/tmp/restricted-kubeconfig
# should print "no" (deletecollection is NOT granted)
oc auth can-i deletecollection secrets -n "${CLUSTER_NS}" --kubeconfig=/tmp/restricted-kubeconfig

If the second command prints anything other than no, stop and fix the Role — the reproduction will not work otherwise.

8. Run destroy with the pre-fix binary against the restricted kubeconfig

./bin/hcp destroy cluster "${PLATFORM}" --name "${CLUSTER_NAME}" --namespace "${CLUSTER_NS}" \
  --sts-creds $STS_CREDS --role-arn $ROLE_ARN \
  --kubeconfig /tmp/restricted-kubeconfig 2>&1 | tee /tmp/destroy-prefix.log
echo "exit code: ${PIPESTATUS[0]}"

Notes

  • This is a CLI-only change (cmd/cluster/core, cmd/util) — no operator/controller code paths are touched, so no HyperShift operator upgrade/downgrade testing is required.
  • The fix intentionally keeps the openshift.io/destroy-cluster finalizer semantics unchanged when destroyPlatformSpecifics itself fails — only deleteCLISecrets failures are now non-fatal. Don't conflate this with masking other destroy failures; only the secrets-cleanup step should be swallowed.
